How to read this This site is written for future-us. Six months from now, when someone asks “why is auth split into all these files?”, this is the answer. Each change has its own page; read them in order — each one builds on the previous.

What this is#

The auth feature already worked. A code review then surfaced five weaknesses plus a messy folder layout. None were user-visible bugs — every one was a maintainability, testability, or correctness risk. We fixed them as six small, ordered commits, each one the smallest change that fully solves its problem (no over-engineering for a student-sized app).

The starting point — what was wrong#

#WeaknessWhat was wrongFixed in
1Tangled test hooksAuthService took five nullable function parameters that existed only so tests could fake behaviour, and every method branched if (hook != null) between a fake path and a real path.01
2Inconsistent injectionOnly AuthGate accepted injected services; the login and onboarding screens grabbed the global Services singleton directly, so they were impossible to unit-test.02
3One widget, three jobsAuthGate decided the screen, buffered deep links, and rendered — trapping genuinely tricky logic (“hold a profile link until ready”) inside a widget where it could not be unit-tested.03
4Race conditionThe gate’s async resolve could be triggered several times concurrently; a slower, older check could finish last and overwrite a newer correct result. It could also crash by notifying after disposal.04
5Swallowed errorsAuth failures were caught and discarded with no logging, and surfaced as a single generic “Unexpected Error” — undiagnosable and unhelpful.05
—Half-organized treeSome features had folders, some didn’t; auth-gate logic lived under routing/; a widget lived under services/.06

The method we followed (identical for every change)#

  1. Elaborate the problem in plain language, grounded in the real code.
  2. Ladder the options from “naïve” (Level 0) to “smart” (Level 3–4), each with pros, cons, and trade-offs.
  3. Decide — pick the smallest change that fully solves it, without over-engineering for a student-sized project.
  4. Implement, prove with tests, then run flutter analyze.

Recurring rule of thumb: Stop climbing the ladder the moment the problem is solved. Going further (DI frameworks, full repository layers, stream-cancellation operators) was real, but over-engineered for this app’s size.

The SWE principles used (glossary)#

Dependency Inversion DIP
Depend on an abstraction you own, not on a concrete vendor library. → 01, 02
Dependency Injection DI
Be handed your dependencies from outside instead of fetching globals yourself. → 02
Single Responsibility SRP
A class should have one reason to change. → 03
Logic / UI separation
Pure Dart classes are easy to test; widgets should stay thin. → 03, 04
Concurrency correctness
When async tasks overlap, the newest result must win — not the slowest. → 04
Observability
Log the real cause so failures in the wild can be diagnosed. → 05
Cohesion by feature
Group files by the feature they serve; keep folders single-purpose. → 06

Result#

  • All auth logic that matters is now unit-tested without spinning up Flutter or Supabase: the AuthService adapter (via mocktail), and AuthGateModel, ProfileLinkHolder, deep_link_parser, and auth_gate_resolver as pure/headless tests.
  • flutter analyze is clean across lib + test; 20 gate tests pass.
  • The app’s behaviour for the user is unchanged; the internals are now testable, diagnosable, consistent, and organized.
