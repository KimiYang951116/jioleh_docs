Change 01 · Dependency Inversion

AuthService Interface

Testability via abstraction — the Dependency Inversion Principle.

Commit eb018f46 files changed+150 / −204 linesDependency Inversion (DIP)

1. The problem#

AuthService’s constructor took five optional functions that existed only so tests could fake behaviour:

auth_service.dart — beforedart
AuthService({
  SupabaseClient? client,
  Session? Function()? currentSession,
  User? Function()? currentUser,
  Future<UserResponse> Function()? getUser,
  Future<void> Function()? signOut,
});

Consequently every method had to branch on “am I in a test?”:

dart
User? getCurrentUser() {
  if (_currentUser != null) return _currentUser();   // fake path
  return _supabase.auth.currentUser;                 // real path
}

The class mixed two unrelated concerns: (1) what it does (its real job) and (2) how we swap it out for tests (a construction concern). That produced four concrete pains.

#PainExplanation
1SmearThe fake-vs-real check was scattered across every method, interleaving test wiring with production logic.
2Half-real objectsYou could inject a fake currentSession but leave the real getUser — so isSignedIn() is fake while hasValidSession() secretly hits the network. The type system allowed this broken mix.
3Invisible contractAsked “what can AuthService do?”, the constructor answered with plumbing (“a client and four optional functions”), not capabilities.
4Doesn’t scaleAdding one testable method = four edits in three places: the method, a constructor parameter, a field, and an if-branch.

Root cause: one class doing both its job and its own test-swapping.

2. Options we discussed (naïve → preferred)#

LevelApproachProsConsVerdict
0No seam — call Supabase directlySimplest codeUntestable — Supabase.instance is a global needing real initRejected
1Function hooks (the original)Tests run without live SupabaseAll four pains aboveReplaced
2Inject the Supabase auth client (GoTrueClient)One injection point; no if-branchesStill coupled to Supabase’s type; tests mock Supabase’s API; vendor lock-inImprovement only
3Program to an interface you ownNo smear; no half-real objects; self-documenting; scales; DIPOne extra file; interface still names Supabase types (User/Session)CHOSEN
4Full repository layer / codegen mocksMaximum flexibilityOver-engineering for one serviceNot chosen

Why Level 3 maps cleanly onto the four pains:

  • No smear → real logic lives only in SupabaseAuthService; fakes live only in FakeAuthService.
  • No half-real objects → an object is wholly real or wholly fake; you cannot mix.
  • Visible contract → the abstract class is the documentation.
  • Scales → a new capability is one interface method + two implementations.

Bonus: This is the Dependency Inversion Principle — the app depends on an abstraction it owns, not on Supabase. Swapping providers later means writing one new class and changing zero tests.

3. File-placement decision#

Q: should the interface and implementation go in separate folders (domain/ vs data/)?

A: No — keep them side by side in services/ for now. Separate folders pay off at a scale we are not at (many services, multiple impls each, a big team). The boundary is already enforced by two things we have: (1) the type everyone uses (AuthService, never the concrete class) and (2) one wiring point (services.dart).

Key trick: the interface keeps the original filename auth_service.dart, so every existing import '.../auth_service.dart' still resolves to the type — no consumer code changed. Only two places ever construct an AuthService: services.dart and the unit test.

4. The change set (with code)#

auth_service.dart — now the interface + exceptions#

lib/services/auth_service.dartdart
abstract class AuthService {
  // Primitives — each implementation provides these.
  User? getCurrentUser();
  bool isSignedIn();
  Future<bool> hasValidSession();
  Stream<AuthState> authStateChanges();
  Future<void> signInWithGoogle();
  Future<void> signOut();

  // Derived helpers — shared by all impls, written once.
  String? getCurrentUserEmail() => getCurrentUser()?.email;
  String getCurrentUserId() {
    final id = getCurrentUser()?.id;
    if (id == null) throw const NotSignedInException();
    return id;
  }
}

supabase_auth_service.dart — the real impl#

The only auth file that imports Supabase.

lib/services/supabase_auth_service.dartdart
class SupabaseAuthService extends AuthService {
  SupabaseAuthService({SupabaseClient? client}) : _client = client;
  final SupabaseClient? _client;
  SupabaseClient get _supabase => _client ?? Supabase.instance.client;

  @override bool isSignedIn() => _supabase.auth.currentSession != null;

  @override
  Future<bool> hasValidSession() async {
    if (!isSignedIn()) return false;
    try {
      final response = await _supabase.auth.getUser();
      return response.user != null;
    } on AuthException {
      return false;   // expired / deleted
    }
  }
  // ...getCurrentUser, authStateChanges, signInWithGoogle, signOut...
}

services.dart — the single place the app commits to Supabase#

lib/services/services.dartdart
static final AuthService auth = SupabaseAuthService(client: _client);

Other services already declared their field as AuthService, so they needed no change. Using extends AuthService (not implements) lets both impls inherit the derived helpers for free.

5. The test fork — a real decision#

The old test relied on the deleted hooks. Its behaviours (e.g. “hasValidSession returns false when getUser throws AuthException”) are now real-adapter logic inside SupabaseAuthService. Two honest options:

OptionProsCons
A — Drop the micro-tests; rely on a fake in consumer testsNo new dependency; fewer lines; common for thin adaptersLoses direct coverage of the AuthException→false path
B — Keep coverage with mocktail (mock SupabaseClient + GoTrueClient)Keeps exact coverageOne dev dependency; mock-heavy tests

Decision: Option B (chosen by the team).

test/services/auth_service_test.dartdart
class _MockSupabaseClient extends Mock implements SupabaseClient {}
class _MockGoTrueClient extends Mock implements GoTrueClient {}

when(() => client.auth).thenReturn(gotrue);
when(() => gotrue.currentSession).thenReturn(_session);
when(() => gotrue.getUser()).thenThrow(const AuthException('expired'));
await expectLater(auth.hasValidSession(), completion(isFalse));

The old signOut assertions were dropped because an earlier refactor already removed that side-effect from hasValidSession.

6. Before / after#

before
BEFORE — one class, two tangled brains:

   +------------------------------------------+
   |               AuthService                |
   | - _client + 4 fake-hook functions        |
   | every method: if(hook!=null) fake : real |
   +------------------------------------------+
        | real                  | fake
        v                       v
   [ Supabase ]           [ test hooks ]
after
AFTER — contract + two implementations:

                +-------------------------------+
                |  <<abstract>>  AuthService    |
                +-------------------------------+
                      ^                   ^
             extends  |                   | extends
        +----------------------+   +----------------------+
        | SupabaseAuthService  |   |   FakeAuthService    |
        +----------------------+   +----------------------+
                  |                          |
                  v                          v
             [ Supabase ]               [ tests ]

   services.dart picks the real one, once.
   Everyone else depends only on the contract.

7. Result & what it unblocks#

  • flutter analyze clean; auth-service + resolver tests pass.
  • No consumer code changed; the app names Supabase in exactly one line.
  • Unblocked Change 02: with an interface + a ready fake, injecting a fake into the login/onboarding screens becomes trivial.

Note: FakeAuthService was added, then removed under YAGNI (Change 01’s tests use mocktail, not the fake), then re-added in Changes 03/04 when a real consumer test finally needed it. The ladder works in both directions.
